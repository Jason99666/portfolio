### 背景与目标

大模型原生的记忆系统依赖模型自主维护，导致记忆内容不可控、不可审计——你无法知道 Agent 记住了什么、为什么记住、是否记错了。同时，开源社区方案普遍兼容性差，缺乏高精度、可审计、可控的记忆管理方案。

目标是为个人使用者设计一套完全可控且人类可直接干预/审计的长期记忆管理系统。

### 方案设计

核心决策是**纯 Prompt 驱动、零代码依赖**——而非构建独立的记忆管理服务。关键权衡：

- **为什么不用向量数据库？** openclaw等开源架构的记忆系统本就来不断迭代，对于个人用户需要的是轻量化可审计的方案
- **为什么纯 Markdown？** 文件天然可被人类直接阅读、编辑和审计，跨平台兼容，零部署成本——直接契合"可审计"核心诉求
- **为什么不用原生记忆？** 不可控、不可审计，openclaw的记忆方案不如hermes，且只有适合自己的才是最好的

在此基础上，设计了**显式标记协议**：用户在对话中主动标记"决策："、"洞察："等关键信息，后台异步提取，避免 Agent 在任务执行中分心。长期记忆采用 **CAR（Context-Action-Result）格式**，确保经验可复用、可验证：

- **Context（场景）**：当时面临什么问题？处于什么情境？
- **Action（行动）**：采取了什么策略或方法？
- **Result（结果）**：得到了什么结果或教训？

基于上述决策，构建了**参考人类认知模型的 L1/L2/L3 三层记忆体系**：

1. **L1 实时工作记忆（每小时提炼）**：从当前会话的任务上下文中提炼关键信息，写入 `SESSION-STATE.md`，对应人类的短期工作记忆
2. **L2 每日记忆（每晚归档）**：从 L1 中提炼归档，过滤噪音，按日期写入 `memory/YYYY-MM-DD.md`，对应人类从日间经验到夜间巩固的过程
3. **L3 长期记忆（每周萃取）**：将 L2 中反复出现的模式提炼为结构化经验法则（Instincts），写入 `MEMORY.md`，对应人类从经验中形成直觉

利用 OpenClaw Cron 定时触发，结合 Isolated Session 机制，确保每次提炼在独立上下文中执行，杜绝上下文污染。

### 技术架构

```
用户对话（含显式标记：决策/洞察）
    ↓
[第 1 层] 物理日志记录
    - 对话实时写入 .jsonl 文件
    - 不依赖向量数据库，纯文本持久化
    ↓
[第 2 层] Cron 定时调度
    - 每小时:00 → L1 提炼：从会话上下文提炼关键信息，写入 SESSION-STATE.md
    - 每晚 23:05 → L2 归档：从 L1 中过滤噪音，按日期写入 memory/YYYY-MM-DD.md
    - 每周日 23:30 → L3 萃取：从 L2 中提炼 CAR 格式经验法则，写入 MEMORY.md
    - 每晚 23:10 → Session GC：清理过期会话上下文
    ↓
[第 3 层] 记忆持久化（Markdown）
    - L1：SESSION-STATE.md（当前任务状态，随时读写）
    - L2：memory/YYYY-MM-DD.md（每日归档，按时间线回溯）
    - L3：MEMORY.md（长期经验法则，CAR 格式结构化存储）
    ↓
[人类审计] 直接阅读 / 手动编辑 / 审计窗口
```

### 业务结果

- **Agent 自动进化**：Agent 真正具备跨会话的自动记忆与进化能力，无需人工干预即可积累领域知识和工作习惯
- **可审计窗口**：纯 Markdown 文件持久化，人类能直接查看、编辑、删除记忆，建立了完全透明的审计通道
- **零运维成本**：纯 Prompt 方案无需部署额外服务，跨平台开箱即用
- **已开源**：为 OpenClaw 生态提供了一个可复用的三层记忆架构参考实现，MIT 协议发布

### 反思与迭代

Harness 是长出来的，不是设计出来的。只有高强度体验，才能对技术边界和真实需求建立感知。简单性是最好的设计原则——纯 Markdown 方案比向量数据库方案更可靠、更易维护，也更符合"可审计"的核心诉求。
