### 背景与目标

非技术背景学习者阅读 AI 论文时，容易陷入数学推导和实验细节，失去对全局贡献的判断力。论文难读的本质是**知识呈网状结构，但论文以线性方式呈现**——读者缺少一张"导航图"来快速定位核心贡献。

目标是将"读论文"这一隐性能力产品化：让非研究者也能在 2 分钟内建立对一篇论文的全局认知，并生成约 3000 字的标准化解读文章。

### 方案设计

解读文章的生成不是单一动作，而是理解、归纳、表达、润色的递进过程。选择**四阶段流水线（R-P-W-R）**而非单轮 Prompt：

- **Reasoning（解读）**：联网检索论文背景，提取研究动机、核心贡献、创新点与局限性
- **Planning（规划）**：根据论文类型自动生成叙事框架——不是套用模板，而是匹配阅读策略
- **Writing（写作）**：基于前两步的中间产物，生成约 3000 字的科普风格解读
- **Rewriting（优化）**：去"AI味"处理——消除套路化句式，确保读起来像人写的

同时引入**多模型路由策略**：Reasoning 阶段使用火山引擎联网检索获取背景，其余三阶段使用 DeepSeek API 处理大纲生成、长文写作和风格润色，实现成本与效果的最优解。

解读论文的**标准五维框架**，是我个人阅读论文的结构化方法，已内化到 Planning 阶段的 prompt 中：

1. **问题定义**：这篇论文要解决什么问题？为什么这个问题重要？
2. **历史瓶颈**：前人为什么没解决好？卡在哪一步？
3. **核心突破**：这篇论文做对了什么？关键创新点是什么？
4. **机制解释**：这个方法为什么有效？底层直觉是什么？
5. **影响与回响**：在技术范式、工程实践层面留下了什么？后续研究如何延续？

这个框架确保解读不是"论文摘要的改写"，而是把论文放回技术演进的脉络中——读者不仅知道"它做了什么"，更知道"它为什么在那个时间点出现"。

### 技术架构

```
PDF / arXiv 文件
    ↓
[第 1 层] 内容解析
    - PyMuPDF 解析双栏排版
    - 还原文本阅读顺序，剔除排版噪音
    ↓
[第 2 层] R-P-W-R 四阶段流水线
    - Reasoning：火山引擎联网检索，理解论文背景与引用上下文
    - Planning：六维框架规划（问题定义/历史瓶颈/核心突破/机制解释/影响/局限）
    - Writing：DeepSeek API 生成约 3000 字科普风格解读
    - Rewriting：去 AI 味处理，消除套路化句式
    ↓
[第 3 层] 输出
    - Streamlit 面板展示 + 对话问答 + 文章编辑
    - Markdown 导出
    ↓
[自动化扩展] OpenClaw 集成skill
    - 自动下载 PDF → 自动触发流水线 → 完整解读文章 → 飞书多维表格管理，零人工干预
```

### 业务结果

- **效率提升 15-30 倍**：2 分钟即可生成约 3000 字的高可读性标准化解读，相比手动阅读 + 笔记整理（通常 30-60 分钟）
- **全流程自动化**：结合 OpenClaw 实现从论文下载到解读文章生成的零人工干预
- **已开源**

### 反思与迭代

论文难读的本质是"知识网状但论文线性"。未来方向是构建知识图谱，提取概念演进与引用关系，以可视化方式呈现——让读者先看森林，再看树木。
